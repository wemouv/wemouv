Explication pas à pas — tests d’intégration de `VehiculeRepositoryTests.java`

Fichier expliqué :
`wemouv/src/test/java/com/diginamic/wemouv/repository/VehiculeRepositoryTests.java`

Objectif global
- Tester “en intégration” les méthodes du repository Spring Data JPA `VehiculeRepository`.
- “En intégration” ici signifie : on charge un contexte Spring limité à la couche persistence (`@DataJpaTest`), on utilise une base H2 en mémoire (profil `test`), Hibernate crée les tables, puis on exécute de vraies requêtes générées par Spring Data.

Pourquoi on persiste des `VehiculeDeService` ?
- `Vehicule` est la classe racine (héritage JPA JOINED).
- `Vehicule` est une entité concrète, mais dans l’application on manipule surtout ses sous-classes.
- En persistants des `VehiculeDeService`, on vérifie que `VehiculeRepository` sait requêter de façon “polymorphe” (la table `vehicule` + jointures vers `vehicule_service` / `vehicule_perso`).

Ce que valide réellement ces tests
- Que Spring Data JPA arrive à :
  - créer la requête correcte à partir du nom de méthode (`findBy...`)
  - exécuter la requête sur une vraie base H2 en mémoire
  - mapper le résultat vers l’entité `Vehicule` (polymorphe)
  - retourner les bons résultats (taille + propriétés vérifiées)
